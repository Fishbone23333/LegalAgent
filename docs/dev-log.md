# 开发日志

## 2026-05-31

### 开发目标

今天主要围绕产品定位和红蓝对抗后的二次交互能力进行优化。项目名称从偏正式的法律工具定位，调整为更贴近毕业生、应届生和刚步入社会人群的「小白合同避坑助手」；同时补齐红蓝对抗分析结束后的追问能力，让用户可以继续围绕同一份合同分别向红方、蓝方和裁决官提问。

### 修改内容

1. 完成项目前端品牌更名。
   - 将前端展示名称调整为「小白合同避坑助手」。
   - 更新首页标题、导航品牌名和核心定位文案，使产品表达更贴近“签合同前帮小白少踩坑”的目标。
   - 更新分析页标题和侧边栏品牌名，保持首页与工作台页面的一致性。

2. 新增红蓝对抗后的三 Agent 追问功能。
   - 红蓝对抗分析完成后，结果页提供「继续追问」区域。
   - 用户可以在红方 Challenger、蓝方 Defender、裁决官 Judge 三个角色之间切换。
   - 三个 Agent 各自保留独立的页面内聊天记录，便于用户围绕不同立场连续追问。
   - 追问上下文包含原始合同文本、合同类型和本次红蓝对抗分析结果，使回答能接续前面的分析内容。

3. 优化追问 Agent 人设与立场边界。
   - 红方继续站在弱势签约者立场，重点帮助用户发现不利条款和维权切入点。
   - 蓝方改为更明确地站在对方律师、企业法务、房东代理人或中介代理人的视角，优先模拟对方可能如何辩护和谈判。
   - 蓝方不再泛泛给用户建议，而是先维护对方立场；遇到明显违法或实务上难以成立的条款时，再说明辩护边界和可能接受的折中方案。
   - 裁决官保持中立裁判视角，负责区分合法、无效、可谈判和高风险条款，并给出更稳妥的行动建议。

4. 调整追问窗口位置。
   - 将追问模块从红蓝对抗结果底部移动到三个 Agent 总结卡片正下方。
   - 新位置位于风险点模块上方，用户看完红方、蓝方和裁决官总结后可以立即继续提问。
   - 追问模块横跨整行展示，避免分散到三个 Agent 卡片中造成高度不稳定或聊天历史割裂。

### 技术实现要点

- 后端新增红蓝对抗追问流式接口 `/debate/chat/stream`，由追问模块根据用户选择的 Agent 生成对应立场的回答。
- 追问请求携带 `agent`、`question`、`context` 和 `history`，其中 `history` 只传入当前 Agent 的既有对话，避免不同角色之间串话。
- 前端在红蓝对抗结果渲染时维护 `debateChat` 状态，包含当前选中的 Agent、三个角色的独立历史和流式请求状态。
- 追问模块在最终结果中插入到 Agent 总结区与风险点汇总区之间，使交互顺序更符合用户阅读路径。

### 验证结果

- 后端追问模块通过 Python 编译检查。
- 前端内联 JavaScript 语法检查通过。
- 静态检查确认追问功能只在红蓝对抗分析完成后展示，并且位置已移动到三个 Agent 总结卡片下方、风险点模块上方。

### 后续计划

- 在完整后端环境中跑一次真实红蓝对抗分析，并测试三个 Agent 的连续追问体验。
- 继续观察蓝方回答是否足够站在对方代理人立场，同时避免误导用户接受明显违法或严重不公平条款。
- 后续如需要长期保存追问记录，可以再评估是否加入本地缓存或账号级历史记录。

## 2026-05-30

### 开发目标

今天的开发重点是补齐合同分析结果的 PDF 交付能力，并同步修复快速分析中风险识别不稳定、PDF 生成异常以及红蓝对抗结果展示不一致的问题。整体目标是让快速分析和红蓝对抗两条路径都能产出结构清晰、内容范围准确、可下载的报告，同时提升明显合同风险的兜底识别能力。

### 修改内容

1. 完善 PDF 下载功能。
   - 前端在快速分析和红蓝对抗结果顶部加入“下载 PDF”入口，用户完成分析后可以直接导出报告。
   - 后端新增 `/analyze/export-pdf`，并完善 `/debate/export-pdf`。
   - 快速分析 PDF 覆盖原始合同、合同类型、风险点、行动建议等内容。
   - 红蓝对抗 PDF 覆盖原始合同、合同类型、三方总结、风险点、合同修订建议、谈判话术等内容。
   - 清理红蓝 PDF 中旧的裁决行动指南、证据清单等非本次范围模块，使报告内容与当前页面结果保持一致。

2. 修复 PDF 生成中的长文本分页问题。
   - 原始合同文本过长时，ReportLab Table 单元格无法跨页，可能触发 `Flowable too large` 报错。
   - 已将原始合同拆分为多个可分页的小块输出，避免单个表格单元格承载超长文本。

3. 修复快速分析偶发“未发现明显风险点”的问题。
   - 增强 `risk_checker` 的 JSON 解析逻辑，兼容模型返回中的 `<think>` 内容。
   - 标准化 `risks` 与 `action_plans` 输出，减少模型返回字段不稳定导致的结果丢失。
   - 增加规则兜底识别，覆盖押金过高、提前退租赔偿过高、维修责任过度转嫁、押金退还周期过长、放弃社保、试用期超期、竞业限制无补偿、加班无补偿、违约金过高等明显风险。

4. 优化红蓝对抗谈判话术展示。
   - 将谈判话术 UI 从两列卡片调整为一列全宽卡片。
   - 在对应条款前增加“对应条款：”标签，使页面视觉更贴近 PDF 报告结构。

### 技术实现要点

- PDF 导出接口拆分为快速分析和红蓝对抗两条路径，分别组织各自需要的报告内容，避免复用旧模块造成信息范围混乱。
- PDF 原始合同区域不再依赖单个超长 Table 单元格，而是按文本块拆分后交给 ReportLab 分页排版。
- `risk_checker` 在 LLM 输出解析前先清理思考标签和 Markdown 包裹，再对结构化字段做类型标准化。
- 明显风险识别采用规则兜底补充模型结果，保证高确定性条款不会因为模型输出格式异常而被漏报。
- 前端谈判话术区域采用单列全宽布局，并与 PDF 报告中的字段命名保持一致。

### 验证结果

- `python -m py_compile backend\api\main.py backend\pdf_generator.py backend\agent\risk_checker.py` 通过。
- 前端内联 JavaScript 语法检查通过。
- 隔离测试验证租赁合同兜底风险能够识别 4 类风险。
- 当前本地 Python 环境缺少 `reportlab` 和 `langgraph`，因此尚无法在该环境运行完整后端真实 PDF 生成和完整工作流。

### 后续计划

- 在依赖完整的后端环境中补跑快速分析和红蓝对抗的真实 PDF 导出流程。
- 继续用真实劳动合同和租赁合同样本验证规则兜底与 LLM 风险结果合并后的展示质量。
- 检查前端 PDF 下载按钮在失败、重复点击和分析结果为空等边界状态下的提示体验。

## 2026-05-28

### 开发目标

今天主要围绕前端分析体验进行优化，重点是让「快速分析」和「红蓝对抗分析」的结果展示更清晰，减少用户误触和页面跳动，并提升红蓝对抗实时生成过程的可读性。

### 修改内容

1. 调整快速分析与红蓝对抗分析的结果模块分工。
   - 快速分析结果中保留「风险点汇总」和「行动优先级」。
   - 移除快速分析结果中的「合同修订建议」和「谈判话术」模块。
   - 红蓝对抗结果中移除「裁决行动指南」模块。
   - 将红蓝对抗结果中的对应位置替换为「合同修订建议」。
   - 将「可复制谈判话术」改为普通「谈判话术」模块。

2. 同步修改分析流程进度块文案。
   - 快速分析最后一步改为 `ActionPlan 行动建议`。
   - 红蓝对抗中 `Judge` 步骤改为「裁决与修订方向」。
   - 红蓝对抗中 `Negotiation` 步骤改为「谈判话术」。
   - 后端流式进度消息也同步调整，避免运行时出现旧文案。

3. 修复快速分析完成后结果不显示的问题。
   - 问题原因是完成后的状态收尾逻辑可能把刚渲染的结果重新覆盖成旧缓存。
   - 修复方式是改用结果 HTML 和进度判断完成态，避免依赖中文状态字符串比较。

4. 优化红蓝对抗实时生成展示。
   - 将原来单独的「实时生成」日志卡片移除。
   - 改为把 `challenger`、`defender`、`judge` 的实时生成内容分别显示在对应 Agent 卡片内。
   - 红方标题使用红色，蓝方标题使用蓝色，裁决官标题使用黄色。

5. 修复红蓝对抗实时生成时的页面滚动问题。
   - 生成过程中 Agent 卡片固定高度，内容在卡片内部滚动，避免页面布局被实时内容撑高。
   - 停止强制拉动整个页面到底部，用户手动上拉时不会再被弹回。
   - 增加用户滚动状态检测，减少自动滚动对阅读的干扰。

6. 增加红蓝对抗最终结果的顺序渐显效果。
   - 最终结果按顺序出现：红方总结、蓝方总结、裁决官总结、风险点汇总、合同修订建议、谈判话术。
   - 通过 CSS 动画和延迟控制实现渐显，不改变后端接口。

### 技术实现要点

- 主要修改文件：`frontend/analysis.html`。
- 后端流式文案同步修改文件：
  - `backend/api/main.py`
  - `backend/agent/debate_streaming.py`
- 红蓝对抗实时内容根据 `event.agent` 分流到对应卡片。
- 最终结果展示通过 `reveal-item` 和 `--reveal-delay` 控制出现顺序。
- 生成态 Agent 卡片使用固定高度和内部滚动，避免布局抖动。

### 验证结果

- 前端内联 JavaScript 语法检查通过。
- 后端修改文件通过 Python 编译检查。
- 静态检查确认旧的「裁决行动指南」「可复制谈判话术」前端展示已移除。

### 后续计划

## 2026-05-27

### 开发目标

这一天主要完成项目重新设计后的前端替换与核心交互流程修复。重点是把 OpenDesign 设计稿落到项目页面中，同时理顺 OCR、快速分析、红蓝对抗分析之间的状态关系，避免页面误跳转、误分析和结果互相覆盖。

### 修改内容

1. 替换和整合前端页面。
   - 将 OpenDesign 文件夹中的重新设计页面合入项目现有前端。
   - 保留项目原有后端接口调用能力，并将新页面接入快速分析、红蓝对抗分析和 OCR 流程。
   - 调整页面结构，让输入区、进度区、结果区和红蓝对抗区形成统一工作台体验。

2. 优化合同类型识别交互。
   - 去掉原来的合同类型手动选择按钮。
   - 在原位置改为提示语，说明系统会自动识别劳动合同或租赁合同。
   - 将输入框示例内容改为浅灰色 placeholder，避免用户误以为示例文本是真实输入。

3. 调整 OCR 识别流程。
   - 点击 OCR 上传图片后立即开始识别，不再要求用户额外点击一次开始识别。
   - OCR 识别完成后自动把识别文本填入合同输入框。
   - OCR 识别后不再自动开始风险分析，用户可以先检查文本，再手动点击分析按钮。
   - 增加 OCR 识别进度条，平时隐藏，识别时显示。

4. 修复 OCR 与分析模式的关系。
   - 原来 OCR 后会自动跳转到红蓝对抗分析，改为在哪个模式上传 OCR，就留在哪个模式。
   - OCR 结果只负责填充输入框，不改变用户当前选择的分析模式。

5. 清理未分析时的默认结果展示。
   - 去掉页面初始状态下的示例风险点、谈判话术和三个 Agent 观点。
   - 改为只有分析完成后才显示真实结果，避免用户误解为已经分析过。

6. 修复快速分析进度和结果展示。
   - 修复快速检查进度块不正常的问题。
   - 将“确认文本属于劳动或租赁合同”改为“确认合同类型”。
   - 调整快速分析进度逻辑，避免进度到 100% 后结果还要等待较久才出现。
   - 改为最终结果准备好后再显示 100%，让 100% 和结果出现保持一致。

7. 隔离快速分析和红蓝对抗分析的页面状态。
   - 修复快速分析过程中切到红蓝对抗，再切回快速分析时页面被重置的问题。
   - 为不同分析模式保存独立的输入、进度、步骤、结果和按钮状态。
   - 支持某个模式分析中切换到另一个模式，切回来后仍能看到原来的分析过程或分析结果。

8. 优化风险点展示。
   - 将快速分析中的风险点统一放入“风险点汇总”模块。
   - 风险点按严重程度从高到低排序。
   - 改善风险点展示结构，避免像普通段落一样缺少标题和分组。

9. GitHub 上传与敏感文件处理。
   - 梳理项目上传到个人 GitHub 的命令流程。
   - 讨论 `.env` 文件和敏感密钥泄露风险。
   - 明确上传前需要检查 `.env` 是否被 Git 跟踪，避免再次推送敏感信息。
